## What We Have Right Now

I went through everything related to notifications in the codebase and heres the current state.

### Already Working
- **expo-notifications** (v0.32.13) and **expo-device** (v8.0.9) are installed
- `usePushNotifications` hook exists at `src/features/notifications/logic/usePushNotifications.ts`
- The hook registers the device, requests permissions, creates an Android notification channel, and gets an Expo push token
- The hook handles notification responses with deep linking support (notifications can include a `route` in their data to navigate the user)
- The parent home screen (`app/(parent)/(tabs)/home.tsx`) already calls `usePushNotifications()` and has the `expoPushToken` available
- Notification settings UI exists for both parent and teacher with toggles for push, messages, and announcements stored in AsyncStorage
- `app.json` has the EAS project ID configured: `5774160a-4a30-46e1-af05-15f5cdc61e87`
- The `expo-notifications` plugin is enabled in app config

### Whats Missing
1. **No push token storage** - the Expo push token gets generated on the device but never gets saved anywhere on the backend. Without storing it theres no way to look up a users token to send them a notification
2. **No server side trigger** - when someone sends a message nothing happens on the backend to trigger a push notification to the recipient. Messages only show up if the other person has the conversation open (via AppSync subscription)
3. **No Lambda function** - theres no AWS Lambda in the Amplify backend at all. We need one to listen for new messages and fire off notifications
4. **Notification preferences are not checked** - the settings screen saves preferences to AsyncStorage but nothing actually reads those preferences before sending a notification
5. **No SNS or Pinpoint** - no AWS notification service is configured

## How Push Notifications Work in Our Stack

Heres the flow from start to finish for how this needs to work:

```
User sends message
        |
        v
createMessage mutation hits DynamoDB
        |
        v
DynamoDB Stream triggers Lambda function
        |
        v
Lambda reads the new message, figures out the recipient
        |
        v
Lambda queries PushToken table for recipients Expo push token
        |
        v
Lambda calls Expo Push API with the token and notification payload
        |
        v
Expo Push API routes to APNs (iOS) or FCM (Android)
        |
        v
Recipient device shows the notification
        |
        v
User taps notification, app opens to the conversation
```

We are using **Expo Push API** as the notification gateway instead of directly integrating with SNS or Pinpoint. The reason is we already have Expo push tokens being generated by the existing `usePushNotifications` hook. Expo handles the routing to APNs and FCM for us. This is way simpler than setting up SNS topics or Pinpoint campaigns for what is essentially just "notify this one person that they got a message."

## The Implementation Plan

### Step 1: Push Token Storage (DynamoDB + Schema)

We need a place to store push tokens. Adding a new model to the GraphQL schema.

**File:** `amplify/backend/api/projectclara/schema.graphql` - EDIT

```graphql
type PushToken @model @auth(rules: [{ allow: public }]) {
  id: ID!
  userId: ID! @index(name: "byUser", queryField: "pushTokensByUserId")
  userType: SenderType!
  token: String!
  platform: String!
  createdAt: AWSDateTime!
  updatedAt: AWSDateTime!
}
```

This gives us:
- One row per device per user (a user could have multiple devices)
- `byUser` index so we can quickly query "give me all push tokens for user X"
- `userType` (PARENT or TEACHER) so we know what kind of user this is
- `platform` (ios or android) in case we ever need platform specific logic

After adding this we run `amplify push --yes` and `amplify codegen` to generate the queries and types.

### Step 2: Token Registration (Frontend)

**New file:** `src/features/notifications/api/pushTokenRepo.ts`

This file handles saving and cleaning up push tokens in DynamoDB. Two functions:

- `registerPushToken(userId, userType, token, platform)` - saves the token. Checks if a token already exists for this user+device combo first to avoid duplicates
- `unregisterPushToken(userId)` - removes the token when the user logs out

Follows the `RepoResult<T>` pattern like everything else in the app.

**Modified file:** `src/features/notifications/logic/usePushNotifications.ts`

After the hook gets the Expo push token it needs to save it to DynamoDB. The hook already runs on mount so we just add the registration call after the token is obtained. We also need to pass in the userId and userType so it knows who this token belongs to.

**Modified files:** `app/(parent)/(tabs)/home.tsx` and `app/(teacher)/(class)/(tabs)/class.tsx` (or wherever the teacher hook runs)

Pass the userId and userType to usePushNotifications so it can register the token for the correct user.

### Step 3: Lambda Function for Message Notifications

**New directory:** `amplify/backend/function/messageNotification/`

This is an AWS Lambda function that triggers when a new message is created. Amplify lets us add Lambda functions that respond to DynamoDB streams.

The Lambda does the following:
1. Receives the DynamoDB stream event containing the new message
2. Extracts the `conversationId` and `senderId` from the message
3. Queries the Conversation table to get the conversation details (parentId, teacherId, parentName, teacherName)
4. Figures out who the recipient is (if sender is the parent, recipient is the teacher and vice versa). For GROUP conversations, gets all parents in the class
5. Queries the PushToken table using the recipients userId to get their Expo push tokens
6. Sends the notification via Expos Push API (`https://exp.host/--/api/v2/push/send`)

The notification payload looks like this:

```json
{
  "to": "ExponentPushToken[xxxxxx]",
  "title": "New message from Mrs. Smith",
  "body": "Hey, just wanted to let you know...",
  "data": {
    "route": "/(parent)/conversation",
    "conversationId": "abc-123",
    "conversationTitle": "Mrs. Smith"
  },
  "sound": "default",
  "badge": 1
}
```

The `data.route` field is already supported by our existing notification handler in `usePushNotifications`. When the user taps the notification it will navigate them to the conversation.

To add this Lambda through Amplify:

```bash
amplify add function
# Name: messageNotification
# Runtime: Node.js
# Trigger: DynamoDB Stream on Message table
```

### Step 4: Respect Notification Preferences

The notification settings screen already saves preferences to AsyncStorage with this shape:

```json
{ "push": true, "messages": true, "announcements": true }
```

But these are stored locally on the device. The Lambda running in AWS cant read AsyncStorage. We have two options:

**Option A (simpler):** Add a `notificationPrefs` field to the Parent and Teacher models in the schema. When the user toggles a preference in settings, save it to DynamoDB as well as AsyncStorage. The Lambda checks this field before sending.

**Option B (local only):** Let all notifications come through from the server but filter them on the device using the local preferences. The `usePushNotifications` hook can check AsyncStorage when a notification arrives and suppress it if the user has that type disabled.

I think **Option B** is the better move for now. Its less schema changes, no migration needed, and it still gives the user control. The downside is the server still sends the notification (wasted bandwidth) but for our scale thats negligible. We can always move to Option A later if we need to.

### Step 5: Teacher Side Token Registration

The teacher side needs the same token registration. Right now `usePushNotifications` only runs on the parent home screen.

**Modified file:** Teacher layout or class screen needs to call `usePushNotifications` with the teachers userId and userType so their token gets registered too.

### Step 6: Announcement Notifications

Since we already have the announcements feature working with DynamoDB, we can extend the Lambda (or create a second one) to trigger on new announcements. When a teacher posts an announcement:

1. Lambda triggers on Announcement table stream
2. Gets the classId from the announcement
3. Queries enrollments by classId to get all studentIds
4. Queries students to get parentIds
5. Queries PushToken table for all those parents
6. Sends batch notifications via Expo Push API

This reuses the same infrastructure we build for message notifications.

## What Needs To Happen With AWS

Before we can deploy the Lambda function we need:
- AWS credentials configured (amplify configure)
- Permission to add a Lambda function to the Amplify project
- The Lambda needs IAM permissions to read from the Conversation and PushToken DynamoDB tables
- The Message table needs DynamoDB Streams enabled (Amplify handles this when you configure the trigger)

## Testing Strategy

### Token Registration
- Login as parent debug, check DynamoDB PushToken table for the token entry
- Login as teacher debug, check same table
- Login on a second device, verify a second token row appears for the same user
- Logout and verify token is removed

### Message Notifications
- Parent sends message to teacher while teacher app is closed. Teacher should get a push notification
- Teacher sends message to parent while parent app is closed. Parent should get a push notification
- Parent sends message while teacher has the conversation open. Teacher should NOT get a duplicate (they already see it via subscription)
- Test with notification preferences toggled off. Notification should be suppressed on the device

### Deep Linking
- Tap a message notification. App should open directly to that conversation
- Tap an announcement notification. App should open to the announcements screen

### Edge Cases
- User with no push token (denied permissions). Lambda should handle gracefully, no crash
- User logged into multiple devices. Both devices should get the notification
- Group conversation. All parents in the class should get notified except the sender
